# ID Conventions

Reference for entity ID prefixes, generation rules, and naming conventions across the Firm framework.

## Core Concepts

### ID Prefix Registry

| Entity | Table | Prefix | Example |
|--------|-------|--------|---------|
| Firm | firm | (user-defined) | chrisai |
| Member | member | MEM | MEM-001 |
| Operation | operation | OPS | OPS-001 |
| Project | project | PROJ | PROJ-001 |
| Unit | unit | UNIT | UNIT-001 |
| Unit (sub) | unit | SUB | SUB-001 |
| Gate | gate | GATE | GATE-001 |
| Goal | goal | GOAL | GOAL-001 |
| Comment | comment | COM | COM-001 |
| Contract | contract | CON | CON-001 |
| Document | document | DOC | DOC-001 |
| Records | records | LOG | LOG-001 |
| Usage Event | usage_event | USG | USG-001 |
| Firm Secret | firm_secret | KEY | KEY-001 |

### Generation Rule

All IDs follow the pattern `{PREFIX}-{NNN}`:
- `NNN` = `COUNT(*) + 1` for rows with matching `firm_id` in the entity's table
- Zero-padded to 3 digits: `MEM-001`, `MEM-012`, `MEM-123`
- Generated by `firm.services._id.next_id(conn, table, prefix, firm_id)`

**Special cases:**
- **Firm ID** is user-defined (e.g., `chrisai`), not auto-generated
- **Sub-Units** use `SUB-NNN` prefix when `parent_unit_id` is set, `UNIT-NNN` otherwise
- **Records** and **Usage Event** IDs are generated by Phase 2 handlers (`_next_records_id`, `_next_usage_id`)

### Concurrency Note

COUNT-based generation is not concurrency-safe. v1 is single-operator, so this is acceptable. If the framework grows concurrent writers (Phase 6 MCP), switch to a sequence table or UUID scheme.

## Anti-Patterns

| Pattern | Why It's Wrong |
|---------|---------------|
| Hardcoded IDs in service code | Always use `next_id()` - manual IDs break sequence |
| Skipping prefix for entity type | Every entity has a prefix. No bare integers. |
| Reusing a deleted entity's ID | IDs are append-only. Deleted rows still count toward sequence. |
| Using UNIT prefix for sub-units | Sub-units use SUB prefix to distinguish nesting level |
