<objective>
## Goal
Build the Python package scaffold + SQLite foundation for the agent-company-architecture framework. After this plan, running `python -m firm init <workspace>` creates a `.firm/firm.db` with a migration-tracking table, ready for entity schemas in Plan 01-02.

## Purpose
The framework needs a working Python package + database substrate before any entity work makes sense. This plan establishes the runtime foundation (SQLite + migration runner) that every subsequent plan builds on. Ships the "scaffold + DB init" shape without locking in any entity schemas yet.

## Output
- Python package at `apps/agent-company-architecture/src/firm/`
- Working `firm init` CLI that initializes `.firm/firm.db` at a workspace path
- Migration runner with a `_migrations` tracking table
- pytest harness with passing smoke tests
</objective>

## AC-1: Package installs and is CLI-invokable
```gherkin
Given a fresh clone of apps/agent-company-architecture/
When `pip install -e .` runs from the package root
Then the `firm` Python package is importable
  And `python -m firm --help` prints help listing the `init` subcommand
  And `python -c "import firm; print(firm.__version__)"` prints a version string
```

## AC-2: `firm init` creates SQLite DB with migrations applied
```gherkin
Given a workspace directory without a .firm/ folder
When `python -m firm init <workspace>` is invoked
Then .firm/ is created at the workspace path
  And .firm/firm.db exists as a valid SQLite database
  And the _migrations table contains one row for 001_init
  And the command exits 0
```

## AC-3: Re-running init is idempotent
```gherkin
Given a workspace that already has .firm/firm.db
When `python -m firm init <workspace>` is invoked again
Then the command exits 0 with an "Already initialized" message
  And the existing DB is not modified
  And no duplicate migration rows are added
```

## AC-4: Migration runner applies pending migrations in numeric order
```gherkin
Given a migrations/ directory with numbered .sql files
When apply_migrations() is invoked against a connected DB
Then all unapplied migrations are applied in numeric order
  And each applied migration is recorded in _migrations
  And subsequent runs with the same migrations return an empty applied list
  And failures inside a migration roll back that migration's transaction
```

  <action>
    Implement the SQLite foundation using stdlib `sqlite3`.

    src/firm/core/db.py:
    - `get_db_path(workspace: Path) -> Path`: returns `workspace / ".firm" / "firm.db"`.
    - `connect(db_path: Path) -> sqlite3.Connection`: opens a connection with `PRAGMA foreign_keys = ON`, `row_factory = sqlite3.Row`. Creates parent dir if missing.
    - `db_connection(workspace: Path)` context manager: yields a connection; commits on clean exit, rolls back on exception, always closes.

    src/firm/core/migrate.py:
    - `MIGRATIONS_DIR` resolved at import time via `importlib.resources` or `Path(__file__).parent.parent / "migrations"` (src-layout-safe).
    - `discover_migrations(migrations_dir: Path) -> list[tuple[int, str, Path]]`: returns sorted (number, name, path) tuples for files matching `^\d{3}_[a-z0-9_]+\.sql$`.
    - `ensure_migrations_table(conn)`: runs `CREATE TABLE IF NOT EXISTS _migrations (id INTEGER PRIMARY KEY, name TEXT UNIQUE NOT NULL, applied_at TEXT NOT NULL DEFAULT (datetime('now')))`.
    - `applied_migration_names(conn) -> set[str]`: returns the set of already-applied migration names.
    - `apply_migrations(conn: sqlite3.Connection, migrations_dir: Path | None = None) -> list[str]`: ensures the table, computes pending, applies each pending migration inside `BEGIN; ... COMMIT;` (rollback on exception), records in `_migrations`, returns applied names in order.

    src/firm/migrations/001_init.sql:
    - Creates `_migrations` table (idempotent via IF NOT EXISTS — the runner also ensures this, so 001_init being a no-op here is fine; include a harmless SELECT 1 comment if needed to keep the file non-empty, OR put the `_migrations` CREATE here and let ensure_migrations_table be a belt-and-suspenders safety net).
    - Decide: have 001_init own the `_migrations` table creation. ensure_migrations_table() runs FIRST (before applying 001_init) so the runner can even track that 001_init itself was applied. This bootstrap ordering is the single subtle point — comment it clearly.
    - No entity tables here. Those land in Plan 01-02.

    tests/test_db.py (pytest, use `tmp_path` fixture):
    - `test_get_db_path_returns_expected_layout`: assert path structure.
    - `test_connect_enables_foreign_keys`: query `PRAGMA foreign_keys` returns 1.
    - `test_connect_uses_row_factory`: fetchone returns sqlite3.Row.
    - `test_db_connection_rolls_back_on_exception`: insert in a `with` that raises, verify row is not persisted.

    tests/test_migrate.py:
    - `test_fresh_db_applies_001_init`: fresh DB → apply_migrations returns ["001_init"] and _migrations has one row.
    - `test_rerunning_is_noop`: second apply_migrations returns [].
    - `test_discover_migrations_sorts_by_number`: write fixture migrations 002_foo.sql, 010_bar.sql, 003_baz.sql to a tmp dir → discover returns them in order 002, 003, 010.
    - `test_migration_failure_rolls_back`: write a fixture migration with invalid SQL → apply raises, _migrations does NOT contain the failed migration, DB state is unchanged.

    Avoid: external migration libraries (alembic, yoyo), rollback-down support (not needed for v1), auto-discovery outside the numbered-file pattern.
  </action>

    Implement the `firm init <workspace>` command end-to-end.

    src/firm/cli/init.py:
    - `run_init(workspace: Path, force: bool = False) -> int` (returns exit code).
    - Expand user (`~`) and absolutize the workspace path.
    - If workspace does not exist: print error to stderr, return 1.
    - Compute db_path via get_db_path().
    - If db_path exists and not force: print `"Already initialized: <db_path>"` to stdout, return 0 (idempotent path — do NOT modify DB).
    - Otherwise: create .firm/ dir, open connection via db_connection, call apply_migrations.
    - Print `"Initialized .firm/firm.db at <workspace>"` and the list of applied migrations.
    - Return 0 on success.

    src/firm/__main__.py (update from Task 1 stub):
    - Replace the init stub with a call to `firm.cli.init.run_init(args.workspace, args.force)`.
    - Argparse: `init` subparser with positional `workspace` (Path type) and optional `--force` flag.
    - Entry function `main(argv: list[str] | None = None) -> int` returns the exit code.

    tests/test_init.py:
    - `test_fresh_init_creates_db_with_migrations`: tmp_path workspace → run_init → assert .firm/firm.db exists, _migrations has 001_init row.
    - `test_init_is_idempotent`: run_init twice → second run returns 0 with "Already initialized" message, migration row count unchanged.
    - `test_init_missing_workspace_returns_1`: point at a non-existent path → assert return 1.
    - `test_init_force_reruns_migrations_cleanly`: run_init once, then with force=True (document that force currently just proceeds — for v1 this still short-circuits because the DB is already migrated; acceptable — tests document observed behavior).
    - Capture stdout/stderr via capsys to assert messages.

    Avoid: network calls, hardcoded paths, destructive behaviors on --force (do NOT delete the DB; force only bypasses the "already initialized" short-circuit).
