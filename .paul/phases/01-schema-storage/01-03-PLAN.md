## Goal
Add a generic CRUD repository on top of the 14 entity tables, plus Unit-specific operations (atomic checkout + dependency cycle detection). After this plan, application code can create/read/update/list/delete entities without hand-writing SQL, JSON columns serialize transparently, `updated_at` auto-touches on write, and concurrent Unit claims resolve deterministically (only one wins).

## Purpose
This is the last piece of Phase 1's data substrate. Hooks (Phase 2), slash commands (Phase 3), and the MCP server (Phase 6) all need a single place to go for entity persistence — this module is that place. Atomic checkout is explicitly called out as a project-level requirement in PROJECT.md; cycle detection is required by ENTITY-DESIGN.md's Unit spec.

## AC-2: JSON columns round-trip transparently
```gherkin
Given a member with `suggested_skills` = ["humanizer", "blog:write"]
When `repo.create` is called with the list in a Python list
Then the stored value in the DB is a JSON string
  And `repo.get` returns the field back as a Python list equal to the original
```

## AC-3: `updated_at` auto-touches on update; immutable tables reject updates
```gherkin
Given a row with a known `updated_at` value
When `repo.update` is called (even for a no-op field set)
Then the stored `updated_at` is later than before
  And when `repo.update` is called against an immutable table (comment, records, usage_event)
  Then sqlite3.IntegrityError is raised (from the existing trigger)
```

## AC-4: Atomic Unit checkout resolves deterministically
```gherkin
Given a Unit row with `claimed_by IS NULL`
When `units.checkout(conn, "UNIT-001", "MEM-001")` is called
Then it returns the updated row dict with `claimed_by = "MEM-001"` and `claimed_at` populated
  And the Unit's status transitions from 'pending' to 'in_progress'
  And a subsequent `units.checkout(conn, "UNIT-001", "MEM-002")` returns None (already claimed)
  And `units.release(conn, "UNIT-001")` clears `claimed_by` and `claimed_at`
```

## AC-5: Unit depends_on cycle detection prevents cycles
```gherkin
Given Units UNIT-A, UNIT-B, UNIT-C with UNIT-B.depends_on = ["UNIT-A"] and UNIT-C.depends_on = ["UNIT-B"]
When `units.validate_no_cycle(conn, "UNIT-A", ["UNIT-C"])` is called (which would create A→C→B→A)
Then a `CycleError` is raised naming the cycle path
  And when non-cyclic dependencies are passed, the call returns cleanly
  And `units.create(conn, {...depends_on: [...cycle...]})` raises before the row is written
```

  <action>
    Create `src/firm/core/repo.py` — a single module with generic CRUD helpers that work across the 14 entity tables.

    **Public API (all functions take conn first):**

    - `create(conn, table: str, data: dict) -> dict` — inserts a row, returns the inserted row (dict, JSON fields deserialized). Raises on violations.
    - `get(conn, table: str, id: str) -> dict | None` — fetches by primary key; returns None if not found.
    - `update(conn, table: str, id: str, data: dict) -> dict | None` — updates named fields + touches `updated_at`; returns the updated row or None if id not found.
    - `list(conn, table: str, **filters) -> list[dict]` — equality filters via kwargs; returns rows ordered by created_at (stable). Supports None filters (matches NULL).
    - `delete(conn, table: str, id: str) -> int` — deletes by id; returns rowcount (0 or 1). Immutable tables raise from their trigger.

    **JSON column registry:**

    Define a module-level constant mapping table → set of column names that store JSON:
    ```python
    JSON_COLUMNS = {
        "firm": {"operator", "core_values", "partners"},
        "contract": {"runtime_config", "skill_loadout", "domain_loadout"},
        "member": {"suggested_skills", "suggested_domains", "budget"},
        "goal": {"metric"},
        "operation": {"goal_ids", "acceptance_criteria", "project_ids"},
        "project": {"goal_ids", "acceptance_criteria", "unit_ids", "tags"},
        "unit": {"goal_ids", "acceptance_criteria", "depends_on", "outputs", "tags"},
        "member_run": {"usage_event_ids", "outputs"},
        "records": {"details"},
        "firm_secret": {"used_by_member_ids"},
        # tables with no JSON columns omitted
    }
    ```

    **Serialization:** on write, serialize JSON columns via `json.dumps` when the caller passes a list/dict. Leave as-is if the caller already passed a string (allows passing pre-serialized). On read, attempt `json.loads` for each JSON column; if the stored value is NULL, return None; if parsing fails, return the raw string (don't crash — the DB value is authoritative).

    **`updated_at` auto-touch on update:** always append `updated_at = datetime('now')` to the SET clause (but only for tables that have an updated_at column — the immutable tables don't). Either maintain a set of tables with updated_at, or catch `OperationalError` and retry without it. Prefer the explicit set.

    **Tables without updated_at** (skip auto-touch): `comment`, `records`, `usage_event`. (member_run has updated_at.)

    **Mutable tables allow-list** (for sanity check; raise if caller tries to update an immutable table directly — the trigger would anyway, but a nicer error up front helps):
    ```python
    IMMUTABLE_TABLES = {"comment", "records", "usage_event"}
    ```

    **Required fields:** don't manually validate required fields — the DB's NOT NULL and CHECK constraints do this. Let sqlite3.IntegrityError propagate.

    **Safety — table name injection:** `table` parameter is interpolated into SQL (parameterized queries don't parameterize identifiers). Guard with a hard-coded allow-list of all 14 table names; reject anything else with a `ValueError`. Do the same for filter column names in `list()`.

    **Do NOT implement:**
    - Batch operations (create_many, etc.) — defer to a future plan if needed
    - JOIN queries or eager loading of related entities — defer to hooks/MCP layers
    - Custom query builders — direct SQL is fine for this framework
    - Soft-delete — deletes are hard; immutable tables reject via trigger
    - Optimistic locking via row version — atomic ops use WHERE clauses (see Task 2 for Unit)

    **Tests (`tests/test_repo.py`) — aim for thorough, not exhaustive:**
    - `test_create_and_get_roundtrip` — simple member create → get
    - `test_list_with_filters` — multi-member insert, filter by firm_id
    - `test_list_filter_none_matches_null` — inserting a row with NULL field, filter returns it
    - `test_update_touches_updated_at` — capture before, update, capture after, assert after > before (using a tiny `time.sleep(1.1)` if needed to defeat second-resolution; OR compare via a column that explicitly differs, like updating role and verifying updated_at is not the same value as created_at's INSERTED value)
    - `test_json_column_roundtrip_list` — suggested_skills stored and retrieved as a list
    - `test_json_column_roundtrip_dict` — unit.acceptance_criteria with object-shaped entries
    - `test_json_column_null_roundtrips_none` — unset JSON column returns None
    - `test_update_immutable_table_raises` — attempt to update a comment → IntegrityError (from trigger; make sure the nicer ValueError at the allow-list layer fires FIRST instead — pick one behavior and test for it)
    - `test_delete_returns_rowcount` — delete existing returns 1; delete missing returns 0
    - `test_unknown_table_raises_valueerror` — `repo.create(conn, "haxxor; DROP TABLE member; --", {...})` raises ValueError
    - `test_list_orders_by_created_at` — three creates with tiny sleeps, list returns in insertion order

    Create `src/firm/core/units.py` with Unit-specific domain logic. Builds on `repo.py`.

    **Public API:**

    - `class CycleError(Exception)` — raised when a proposed depends_on would create a cycle. Message names the cycle path (`"UNIT-A → UNIT-C → UNIT-B → UNIT-A"`).
    - `checkout(conn, unit_id: str, member_id: str) -> dict | None` — atomic claim:
      - SQL: `UPDATE unit SET claimed_by = ?, claimed_at = datetime('now'), status = CASE WHEN status = 'pending' THEN 'in_progress' ELSE status END, updated_at = datetime('now') WHERE id = ? AND claimed_by IS NULL RETURNING *`
      - Run in a single statement via `executescript`-free `execute` (RETURNING is supported in SQLite 3.35+ which Python 3.12 ships).
      - If `rowcount == 0` (or `fetchone()` is None), the unit was already claimed or doesn't exist — return None.
      - Otherwise, return the updated row dict (use repo's row-to-dict conversion for JSON fields).
      - Validates `member_id` exists by relying on the FK constraint; if the caller passes a fake member_id, sqlite3.IntegrityError propagates.
    - `release(conn, unit_id: str) -> dict | None` — clears `claimed_by` and `claimed_at`; does NOT revert status (status is orthogonal — use repo.update to change status separately). Touches updated_at. Returns the updated row or None if not found.
    - `validate_no_cycle(conn, unit_id: str, depends_on: list[str]) -> None` — walks each dep's own depends_on (via recursive DB traversal) to check if `unit_id` is reachable. If it is, raise `CycleError` with the discovered path. Clean return on success.
    - `create_with_deps(conn, data: dict) -> dict` — convenience wrapper: calls `validate_no_cycle` first (using data['id'] and data.get('depends_on', [])) then repo.create. Raises CycleError before any write.
    - `set_dependencies(conn, unit_id: str, depends_on: list[str]) -> dict` — validates no cycle, then updates just the `depends_on` column. Returns the updated row.

    **Cycle detection implementation:**

    - Use recursive CTE or Python-side BFS. Python-side BFS is simpler and fine for v1 scale:
      ```python
      def validate_no_cycle(conn, unit_id, depends_on):
          if unit_id in depends_on:
              raise CycleError(f"{unit_id} → {unit_id}")
          stack = [[dep] for dep in depends_on]  # list of paths
          visited_at_root = set()  # avoid re-walking the same subgraph
          while stack:
              path = stack.pop()
              current = path[-1]
              if current == unit_id:
                  raise CycleError(" → ".join([unit_id] + path))
              if current in visited_at_root:
                  continue
              visited_at_root.add(current)
              # fetch current's own depends_on
              row = conn.execute("SELECT depends_on FROM unit WHERE id = ?", (current,)).fetchone()
              if row is None or row[0] is None:
                  continue
              try:
                  child_deps = json.loads(row[0])
              except (json.JSONDecodeError, TypeError):
                  continue
              for child in child_deps:
                  stack.append(path + [child])
      ```
    - Handle the edge case where a dep references a non-existent Unit: treat as no further walk (don't crash; the FK-less soft ref is valid per v1).

    - `test_checkout_returns_none_when_already_claimed` — claim once, claim again with different member → None
    - `test_checkout_preserves_status_when_not_pending` — Unit with status='blocked', checkout does not transition to in_progress (stays blocked, but claim succeeds)
    - `test_checkout_nonexistent_unit_returns_none` — claim a UUID that doesn't exist → None
    - `test_checkout_fake_member_raises` — claim with a non-existent member_id → sqlite3.IntegrityError
    - `test_release_clears_claim` — claim then release; claimed_by and claimed_at are NULL
    - `test_release_does_not_touch_status` — claim (status→in_progress), release; status stays in_progress
    - `test_validate_no_cycle_accepts_simple_chain` — A→B→C (via validate), no error
    - `test_validate_no_cycle_detects_self_loop` — `validate_no_cycle(conn, "UNIT-A", ["UNIT-A"])` → CycleError
    - `test_validate_no_cycle_detects_indirect_cycle` — A→C, B→A, validate("C", ["B"]) → CycleError with path
    - `test_validate_no_cycle_handles_missing_unit_gracefully` — depend on a unit id that doesn't exist → no cycle error (silent pass)
    - `test_create_with_deps_blocks_cycles` — try to create UNIT-X with deps that cycle back → CycleError, no row written
    - `test_set_dependencies_updates_and_validates` — happy path + cycle case

    For checkout concurrency: a real concurrency test would need threads/processes. For v1, smoke-test the UPDATE ... WHERE claimed_by IS NULL pattern by calling checkout twice in sequence (SQLite's transaction boundary is the atomicity guarantee; two concurrent connections behave the same as serial calls for this WHERE pattern).
