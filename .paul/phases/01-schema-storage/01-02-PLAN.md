
## AC-2: Firm-scoped foreign keys enforce with ON DELETE CASCADE
```gherkin
Given the schema applied
When a Member row is inserted with firm_id that does not reference an existing Firm
Then the INSERT fails with a foreign key violation
  And when a Firm is deleted
  Then all rows referencing that firm_id in firm-scoped tables are cascade-deleted
```

## AC-3: Immutable entities reject UPDATE and DELETE
```gherkin
Given a row inserted into comment, records, member_run, or usage_event
When a subsequent UPDATE or DELETE is attempted on that row
Then the operation is rejected (trigger raises)
  And the row remains unchanged
```

## AC-4: CHECK constraints on enum-like status fields
```gherkin
Given the schema applied
When a row is inserted with an invalid status value (e.g., unit.status = 'banana')
Then the INSERT fails with a CHECK constraint violation
  And valid values from ENTITY-DESIGN.md are accepted
```

```gherkin
Given the schema applied

  - firm_id (on every firm-scoped table)
  - parent_entity_type + parent_entity_id (on goal and comment)
  - assignee_member_id, claimed_by, and status (on unit)
  - member_id (on member_run, usage_event)
```

  <name>Task 1: Write migration 002_entities.sql (14 tables + FKs + triggers + indexes)</name>
  <files>

  <action>
    **First: read ENTITY-DESIGN.md end-to-end.** Every entity section has a "Design choices locked" table and a "v1 Schema" JSON example. Those two together define each table's columns, types, constraints, and defaults. The "Storage:" lines are obsolete (SQLite pivot) — ignore them.

    **Translate each of the 14 entities into a CREATE TABLE statement.** Group in the file by concern, in this order:

    1. `firm` — singleton-ish (one row per Firm; `id TEXT PRIMARY KEY`, not FK-scoped)
    2. `contract` — runtime contracts (referenced by Member)
    3. `member` — identity entities (references contract)
    4. `goal` — polymorphic via parent_ref
    5. `operation`, `project`, `unit` — work structure (unit depends on project)
    6. `comment` — polymorphic via parent_ref (immutable)
    7. `member_run`, `usage_event` — execution audit (immutable)
    8. `gate` — board approvals
    9. `records` — audit log (immutable)
    10. `firm_secret`, `document` — supporting

    **Conventions to apply to every table (translate consistently):**

    - **Primary key:** `id TEXT PRIMARY KEY` — IDs carry their prefix (e.g., `'MEM-001'`, `'UNIT-001'`). Don't split prefix from number.
    - **Firm scoping:** Every table except `firm` itself has `firm_id TEXT NOT NULL REFERENCES firm(id) ON DELETE CASCADE`.
    - **Timestamps:** `created_at TEXT NOT NULL DEFAULT (datetime('now'))`. Mutable entities also get `updated_at TEXT NOT NULL DEFAULT (datetime('now'))`. Immutable entities (comment, records, member_run, usage_event) have only `created_at`.
    - **Status enums:** Implement as `CHECK (status IN ('a', 'b', ...))`. Pull the enum set from the "Status lifecycle" or equivalent line in each entity's design table.
    - **Array / object fields** (e.g., `tags`, `outputs`, `skill_loadout`, `acceptance_criteria`, `depends_on`, `goal_ids`): store as `TEXT` containing JSON. Add a comment on each such column: `-- JSON array` or `-- JSON object`. No JSON1 CHECK constraints in v1 — trust application layer for now.
    - **Polymorphic parent_ref** (Goal, Comment): split into two columns: `parent_entity_type TEXT NOT NULL CHECK (parent_entity_type IN ('firm', 'member', 'operation', 'project', 'unit', 'goal'))`, `parent_entity_id TEXT NOT NULL`. No composite FK (SQLite's FK system can't express "depends on one of several tables").
    - **Unit-specific fields needed for atomic checkout** (Plan 01-03 will use these): `claimed_by TEXT REFERENCES member(id)` (nullable — null means unclaimed), `claimed_at TEXT`.

    **Indexes:**
    - `CREATE INDEX idx_{table}_firm_id ON {table}(firm_id);` on every firm-scoped table
    - `CREATE INDEX idx_goal_parent ON goal(parent_entity_type, parent_entity_id);`
    - `CREATE INDEX idx_comment_parent ON comment(parent_entity_type, parent_entity_id);`
    - `CREATE INDEX idx_unit_status ON unit(status);`
    - `CREATE INDEX idx_unit_assignee ON unit(assignee_member_id);`
    - `CREATE INDEX idx_unit_claimed_by ON unit(claimed_by);`
    - `CREATE INDEX idx_unit_project ON unit(project_id);`
    - `CREATE INDEX idx_member_run_member ON member_run(member_id);`
    - `CREATE INDEX idx_usage_event_member ON usage_event(member_id);`
    - `CREATE INDEX idx_usage_event_run ON usage_event(run_id);`

    **Immutability triggers** — for each of `comment`, `records`, `member_run`, `usage_event`, create two triggers:

    ```sql
    CREATE TRIGGER trg_{table}_no_update
      BEFORE UPDATE ON {table}
      FOR EACH ROW
    BEGIN
      SELECT RAISE(ABORT, '{table} is immutable');
    END;

    CREATE TRIGGER trg_{table}_no_delete
      BEFORE DELETE ON {table}
      FOR EACH ROW
    BEGIN
      SELECT RAISE(ABORT, '{table} is immutable');
    END;
    ```

    **Pattern reference — full example for Member:**

    ```sql
    CREATE TABLE member (
        id TEXT PRIMARY KEY,
        firm_id TEXT NOT NULL REFERENCES firm(id) ON DELETE CASCADE,
        name TEXT NOT NULL,
        role TEXT NOT NULL,
        contract_id TEXT REFERENCES contract(id),
        status TEXT NOT NULL DEFAULT 'active'
            CHECK (status IN ('active', 'paused', 'retired')),
        skill_loadout TEXT,  -- JSON array
        reports_to_member_id TEXT REFERENCES member(id),
        created_at TEXT NOT NULL DEFAULT (datetime('now')),
        updated_at TEXT NOT NULL DEFAULT (datetime('now'))
    );

    CREATE INDEX idx_member_firm_id ON member(firm_id);
    ```

    (Adjust column list per ENTITY-DESIGN.md's canonical Member schema — the above is a shape reference, not a drop-in.)

    **File header:** Start the migration with a comment block noting its purpose and source:
    ```sql
    -- 002_entities: Creates the 14 entity tables for the firm framework.
    -- Canonical schema source: ENTITY-DESIGN.md (2026-04-14).
    -- Storage notes in ENTITY-DESIGN.md referencing JSON files are superseded
    -- by the SQLite pivot (2026-04-15); only field lists and constraints apply.
    ```

    Avoid:
    - JSON1 CHECK constraints (e.g., `json_valid(column)`) — defer validation to application layer in v1
    - Composite primary keys (use surrogate id + UNIQUE constraints if needed)
    - Down migrations / revert logic (v1 has no rollback)
    - Seeding any rows (no INSERT INTO in this migration)
  </action>
