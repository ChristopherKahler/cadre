<objective>
## Goal
Implement the three cross-cutting service infrastructure modules (_id, _validate, _records) that all entity service modules depend on. Every entity create/update in 03-04 through 03-07 will import from these.

## Purpose
The service layer bridges skill tasks to the repo CRUD layer. These infrastructure modules handle: ID generation (next_id), validation (FK existence, status enums, polymorphic refs), and audit trail (Records auto-entry). Without them, entity services would duplicate this logic.

## Output
- `src/firm/services/_id.py` — unified ID generation
- `src/firm/services/_validate.py` — shared validation helpers
- `src/firm/services/_records.py` — Records auto-entry
- Tests for all three modules
</objective>

<acceptance_criteria>

## AC-1: next_id generates correct prefixed sequential IDs
```gherkin
Given an empty table and a firm_id
When next_id is called
Then it returns "{PREFIX}-001"
And subsequent calls return "-002", "-003", etc.
And the count is scoped to firm_id (different firms get independent sequences)
```

## AC-2: Validation helpers catch invalid references
```gherkin
Given a migrated database with seeded entities
When require_exists is called with a nonexistent ID
Then it raises ValueError with entity-aware message
When validate_parent_ref is called with an invalid type or missing ID
Then it raises ValueError
When validate_status is called with an invalid status
Then it raises ValueError listing allowed values
```

## AC-3: log_event writes immutable Records entries
```gherkin
Given a migrated database
When log_event is called with event_type, actor, target_ref, and optional details
Then a LOG-NNN record is created in the records table
And the record contains all provided fields
And the ID is sequential per firm_id
```
