<objective>
## Goal
Ship the callable `firm.hooks.unit_completion.on_unit_done()` handler that, when a Unit transitions to `done`, writes an immutable `records` row for the status transition and flips `resolved: true` on any `project.acceptance_criteria` entries whose `resolved_by` points at that Unit. Expose it behind a CLI verb `python -m firm unit complete <unit_id> --member <member_id>` so Phase 3's `/unit:complete` slash command can wrap it.

## Purpose
Second of the three Phase 2 hook surfaces (session-pulse done in 02-02; run-record in 02-04). This is where the framework stops being read-only and starts producing the immutable history (`records`) + project-state rollups that every downstream surface (audit trail, Member dashboards, Gate decisions) depends on. Locks in the write-path for Unit lifecycle events before Phase 3 introduces slash commands that will fire it.

## Output
- `firm.hooks.unit_completion` module with `on_unit_done()` entrypoint (transactional)
- `firm.cli.unit` CLI subcommand `complete` wired into `firm.__main__`
- Unit tests covering 4 fixtures: normal / no-match AC / unit-not-found / project-missing
- CLI tests for happy-path + `--dry-run` rendering
- `.paul/phases/02-hook-layer/02-03-SUMMARY.md` after UNIFY
</objective>

```gherkin
  And if neither: the call returns {"ok": false, "reason": "project-missing", ...}

```

    Create `src/firm/hooks/unit_completion.py` with a single public entrypoint:

      def on_unit_done(
          conn: sqlite3.Connection,
          *,
          firm_id: str,
          unit_id: str,
          member_id: str,
          prior_status: str,
          run_id: str | None = None,
          now: str | None = None,  # ISO string for deterministic tests; default = utcnow
      ) -> dict: ...

    Behavior, in order, inside a single `with conn:` transaction block:
      1. Fetch the unit via `repo.get(conn, "unit", unit_id)`. If None → return
         {"ok": False, "reason": "unit-not-found", "unit_id": unit_id}. NO writes, NO exception.
      2. Fetch the project via `repo.get(conn, "project", unit["project_id"])`. If None →
         return {"ok": False, "reason": "project-missing", "unit_id": unit_id,
         "project_id": unit["project_id"]}. NO writes.
      3. Walk project.acceptance_criteria (already a Python list thanks to repo JSON decode).
         For each entry where `entry.get("resolved_by") == unit_id` AND `entry.get("resolved") is not True`,
         set `entry["resolved"] = True`. Collect the ids of mutated entries into `resolved_ac_ids`.
         (Entries already resolved stay unchanged — idempotent.)
      4. Generate a records id: next `LOG-NNN` via the same ULID-or-sequential convention already used
         in `firm.core.repo` create paths. (If no helper exists, inline a `_next_records_id(conn)`
         private function that does `SELECT COUNT(*) FROM records WHERE firm_id = ?` → LOG-{n:03d}.
         Match whatever 02-02 / Phase 1 is doing for consistency — check `repo.py` first.)
      5. Call `repo.create(conn, "records", {...})` with:
           id=<generated>, firm_id, event_type="unit.status_transition",
           actor_type="member", actor_id=member_id,
           target_entity_type="unit", target_entity_id=unit_id,
           details={"prior_status": prior_status, "new_status": "done",
                    "project_id": unit["project_id"], "resolved_ac_ids": resolved_ac_ids},
           run_id=run_id, timestamp=(now or utcnow-iso)
      6. If resolved_ac_ids is non-empty, call `repo.update(conn, "project", project["id"],
         {"acceptance_criteria": <mutated list>})`. If empty, skip the update entirely (AC-2 no-op).
      7. Return {"ok": True, "records_id": <id>, "resolved_ac_ids": resolved_ac_ids,
                 "unit_id": unit_id, "project_id": unit["project_id"]}.

    Hard rules:
      - Transaction boundary is `with conn:` around steps 5+6. If either raises, the block rolls
        back, caller sees exception. (This is the atomicity backing AC-4 — records cannot exist
        without the matching project AC flip, when one is needed.)
      - Do NOT modify `unit.status` here. Status transitions are the caller's concern (Phase 3
        slash command does `repo.update(conn, "unit", id, {"status": "done"})` separately,
        THEN calls on_unit_done). The function documents this assumption in the docstring.
      - Do NOT redact anything — status transitions carry no user content (per BRIEF §3.2).
      - Pyright-clean: full type hints, docstring, no `Any` returns.

    Register the module by adding `from firm.hooks.unit_completion import on_unit_done` to
    `src/firm/hooks/__init__.py` so callers can `from firm.hooks import on_unit_done`.

    Tests in `tests/hooks/test_unit_completion.py` — use existing fixture helpers from
    `tests/hooks/test_session_pulse.py` (in-memory DB, migrations applied, seed helper).
    Required cases:
      - test_happy_path_flips_matching_ac            → AC-1
      - test_already_resolved_ac_is_idempotent       → AC-1 tail (doesn't double-write)
      - test_no_matching_ac_writes_records_only      → AC-2
      - test_empty_acceptance_criteria_list          → AC-2
      - test_unit_not_found_returns_structured_err   → AC-3
      - test_unit_not_found_writes_nothing           → AC-3 (row-count snapshot)
      - test_project_missing_returns_structured_err  → AC-4
      - test_transaction_rolls_back_on_mid_failure   → AC-4 (force an exception after
        records-create via monkeypatching repo.update, assert records row NOT committed
        after the context manager exits)
      - test_records_details_payload_shape           → explicit schema check on details JSON
      - test_run_id_threads_through                  → records.run_id populated when passed
      - test_now_param_is_deterministic              → timestamp override for golden fixtures
  </action>
