<objective>
## Goal
Produce a research brief that locks Phase 2's injection tag format and hook contracts so the three execute plans (02-02 session-pulse, 02-03 unit-completion, 02-04 run-record) can be written without rediscovery.

## Purpose
Phase 2 is where the framework becomes visible in every Claude Code session. The only open question flagged in ROADMAP + PROJECT for this phase is injection format specifics — what exactly goes in `<active-roster>` vs `<pending-gates>` vs `<goal-health>`, and what runtime contracts the completion/record hooks must honor. Locking these decisions now (before writing code) prevents rewrites across three hooks and matches the project's "hard decisions upstream" pattern from Phase 1.

## Output
`.paul/phases/02-hook-layer/02-01-BRIEF.md` — research brief containing:
1. Injection tag format specs (schema + example rendered output for each of the three tags)
2. Hook contracts (trigger event, input payload, SQL reads/writes, exit behavior)
3. Paperclip-derived architectural signals worth borrowing (and explicit anti-signals)
4. Recommended plan split for 02-02 / 02-03 / 02-04 with task outlines
5. Decisions log entries to record in PROJECT.md on completion
</objective>

## AC-2: Hook contracts are unambiguous
```gherkin
Given the three hooks (session-pulse, unit-completion, run-record)
When a developer reads the brief
Then for each hook they can answer without re-reading any other doc:
  - What Claude Code event triggers it (SessionStart, UserPromptSubmit, Stop, etc.)
  - What input payload it receives (env vars, stdin JSON, file paths)
  - What SQLite reads and writes it performs (named queries, entity writes)
  - What it outputs to stdout/stderr (if any) and how Claude Code consumes it
  - What failure modes are tolerated vs fatal
```
